Sectie 3: Klanten, Prijzen & Commerciële Regels

Klanthiërarchie, prijslagen, en wie wat mag

Klantstructuur

Klanthiërarchie: Organisatie → Vestigingen
  ┌─────────────────────────┐
  │   clients              │
  │─────────────────────────│
  │ id                      │
  │ parent_client_id        │──→ clients (NULL = hoofdorganisatie)
  │ name                    │
  │ client_type             │  "wholesaler" / "retailer" / "consumer"
  │ contact_email           │
  │ logo_url                │  (voor stickers)
  │ sticker_text            │  optioneel extra tekst op stickers
  │ active                  │
  └─────────────────────────┘

  Voorbeeld:
    ├── Interieur Groep NL (parent = NULL)
    │   ├── Vestiging Amsterdam (parent = Interieur Groep NL)
    │   ├── Vestiging Rotterdam (parent = Interieur Groep NL)
    │   └── Vestiging Utrecht (parent = Interieur Groep NL)
    ├── Tapijt Boutique Delft (parent = NULL, standalone)

Prijsstructuur

Twee prijslagen: inkoop (Karpi → klant) en verkoop (klant → consument)
  ┌─────────────────────────────┐
  │  client_purchase_prices   │  Inkoopprijs per klant
  │─────────────────────────────│  (wat de klant aan Karpi betaalt)
  │ id                          │
  │ client_id                   │──→ clients
  │ quality_id                  │──→ qualities
  │ finishing_type_id           │──→ finishing_types (optioneel)
  │ price                       │
  │ valid_from / valid_until    │  prijsperiodes
  └─────────────────────────────┘

  Fallback: als er geen klant-specifieke prijs is,
  geldt de base_price uit qualities.

  ┌─────────────────────────────┐
  │  client_retail_prices     │  Verkoopprijs (voor op sticker)
  │─────────────────────────────│  (wat de klant aan consument vraagt)
  │ id                          │
  │ client_id                   │──→ clients
  │ quality_id                  │──→ qualities
  │ dimension_id                │──→ sample_dimensions
  │ price                       │  door klant bepaald
  │ price_per                   │  "piece" / "m2"
  └─────────────────────────────┘

Commerciële regels

Welke klant mag welke combinatie krijgen
  ┌──────────────────────────────┐
  │  client_product_rules      │
  │──────────────────────────────│
  │ id                           │
  │ client_id                    │──→ clients
  │ quality_id                   │──→ qualities
  │ finishing_type_id            │──→ finishing_types (NULL = alle)
  │ rule_type                    │  "allow" / "deny"
  └──────────────────────────────┘

  Werking:
    - Standaard: klant mag alles wat technisch kan
    - "deny" regel: klant X mag NIET tapijt Y met afwerking Z
    - "allow" regel: klant X mag ALLEEN deze combinaties
    
    Als er minstens 1 "allow" regel is voor een klant,
    dan mag die klant ALLEEN wat expliciet is toegestaan.

Sticker-informatie (voorbereiding voor later)

Per sticker wordt samengesteld uit:

  • Klantlogo → uit clients.logo_url
  • Productinfo → collectie + kwaliteit + kleurcode + afwerking
  • Verkoopprijs → uit client_retail_prices
  • Extra tekst → uit clients.sticker_text

De sticker-generator zelf bouwen we in een latere fase, maar alle benodigde data zit al in het model.
